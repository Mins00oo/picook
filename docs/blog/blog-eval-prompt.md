# 개발 블로그 글감 평가 프롬프트

아래 기준에 따라 내가 제시하는 개발 경험이 블로그 글로 쓸 가치가 있는지 판단해줘.

## 핵심 질문

> "이 글을 쓰지 않으면, 같은 상황에 빠진 다음 사람이 나처럼 삽질해야 하는가?"

## 쓸 만한 글의 조건 (하나 이상 해당하면 쓸 만함)

1. **공식 문서대로 했는데 안 됐다**
   — 공식 문서는 이상적인 환경을 가정한다. 실제 환경에서 부딪힌 차이가 있다면 그게 곧 가치다.

2. **검색했을 때 한국어 자료가 없거나 부실했다**
   — "한글 자료가 없네"라고 느꼈다면 그게 신호다. 영어 문서 보고 해결한 과정을 정리하는 것만으로 충분하다.

3. **선택의 이유가 존재한다**
   — 왜 A 대신 B를 골랐는지, 다른 선택지는 왜 버렸는지. 환경 제약·트레이드오프·비교 판단이 들어가면 평범한 주제도 평범하지 않다.

4. **여러 문서를 조합해야 풀리는 문제였다**
   — 하나의 스택오버플로우 답변으로 끝나지 않고, A 문서 + B 이슈 + C 블로그를 종합해서 해결한 문제. 그 조합 과정 자체가 콘텐츠다.

5. **실패 → 재시도 → 해결의 과정이 있다**
   — 처음 시도가 왜 안 됐는지, 그래서 뭘 바꿨는지의 흐름이 있으면 읽는 사람이 "왜"를 이해할 수 있다.

## 쓰지 않아도 되는 글의 조건 (대부분 해당하면 스킵)

- 공식 문서 튜토리얼을 따라한 것 그대로
- 구글 첫 번째 검색 결과로 바로 해결되는 내용
- 본인의 환경·판단·맥락이 전혀 개입되지 않은 순수 요약
- 이미 같은 내용을 더 잘 정리한 한국어 글이 존재함

## 응답 형식

아래 형식으로 답변해줘:

```
### 판단
쓸 만하다 / 애매하다 / 굳이 안 써도 된다

### 해당 조건
위 조건 중 어디에 해당하는지 (번호로)

### 글의 방향 제안
쓸 만하다면, 어떤 포인트를 중심으로 쓰면 좋을지 한두 줄로

### 제목 후보
- 후보 1
- 후보 2
- 후보 3
```
