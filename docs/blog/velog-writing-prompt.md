# Velog 기술 블로그 자동 점검 프롬프트

---

## 실행 흐름

### 0단계: 기존 블로그 글 확인
- `blog/` 디렉토리에 이미 작성된 md 파일 목록을 확인한다.
- 각 파일의 제목과 핵심 주제를 파악해둔다.
- 이 목록이 이후 중복 판단의 기준이 된다.

### 1단계: 커밋 수집
- `git log --since="3 days ago" --oneline --name-only` 로 최근 커밋을 확인한다.
- 커밋 메시지와 변경된 파일 경로를 기반으로 어떤 작업이 있었는지 파악한다.

### 2단계: 글감 판단

아래 핵심 질문에 기반하여 판단한다.

> "이 글을 쓰지 않으면, 같은 상황에 빠진 다음 사람이 나처럼 삽질해야 하는가?"

#### 쓸 가치가 있는 글의 조건 (하나 이상 해당하면 쓸 만함)

- **공식 문서대로 했는데 안 됐다** — 실제 환경에서 부딪힌 차이가 곧 가치다.
- **검색했을 때 한국어 자료가 없거나 부실했다** — 영어 문서 보고 해결한 과정만으로 충분하다.
- **선택의 이유가 존재한다** — 왜 A 대신 B를 골랐는지, 환경 제약, 트레이드오프가 포함된다.
- **여러 문서를 조합해야 풀리는 문제였다** — 조합 과정 자체가 콘텐츠다.
- **실패 → 재시도 → 해결의 과정이 있다** — "왜"를 이해할 수 있는 흐름이 있다.

#### 쓰지 않아도 되는 글 (대부분 해당하면 스킵)

- 공식 문서 튜토리얼을 따라한 것 그대로
- 구글 첫 번째 검색 결과로 바로 해결되는 내용
- 본인의 환경, 판단, 맥락이 전혀 개입되지 않은 순수 요약

### 3단계: 중복/보충 판단

글감이 확정되면, `blog/` 디렉토리의 기존 글과 비교한다.

- **같은 주제의 글이 이미 있고, 새로운 내용이 없다** → 스킵
- **같은 주제의 글이 있지만, 추가 삽질/해결/변경이 있었다** → 기존 md 파일을 보충 수정
- **완전히 새로운 주제다** → 새 md 파일 작성

보충 수정 시에는 기존 글의 흐름을 유지하면서 새로운 내용을 자연스럽게 추가한다. 
예: 이전에 "삽질 3"까지 있었는데 추가 삽질이 있었다면 "삽질 4, 5"를 추가하고 최종 해결을 업데이트.

### 4단계: 결과물

- **새 글 작성 시**: `blog/YYYY-MM-DD-제목.md` 형식으로 저장
- **기존 글 보충 시**: 해당 md 파일을 직접 수정
- **글감 없을 시**: 아무 파일도 생성하지 않음. 커밋도 하지 않음.

작성 또는 수정 후:
```bash
git add blog/
git commit -m "blog: 제목"
git push origin main
```

## 글 작성 스타일

### 문체

- **"~이다", "~했다" 체 사용** (경어체 "~입니다" 아님)
- 문장을 짧게 끊는다. 한 문장이 2줄을 넘기지 않는다.
- 본인 생각을 자연스럽게 삽입한다: "문득", "이쯤에서", "결론부터 말하면" 등
- 과도한 이모지, 강조, 감탄사는 쓰지 않는다

### 글 구조

```
# [카테고리] 제목 — "~해보자" 또는 "~해결" 형태

## 개요
💡 콜아웃으로 1~3줄 요약 (배경 + 핵심 결론)

## 본문
- 트러블슈팅: 상황 → 에러 → 시도 → 실패 이유 → 해결 순서
- 기술 설명: 왜 필요한지 → 개념 → 코드 → 주의점

## 정리
- 핵심 내용을 표 또는 짧은 bullet으로 요약

## 느낀 점
- 배운 것, 다음에 적용할 것 등 개인적 회고 (2~4줄)
```

### 코드 블록 규칙

- 코드 블록 바로 아래에 짧은 설명을 붙인다
- 에러 메시지는 핵심 부분만 발췌한다 (전체 스택트레이스 X)
- 성공/실패 결과를 명확히 표기: "**결과: 실패.**", "**결과: 성공.**"

### 하지 않는 것

- 개념 설명으로 시작하지 않는다 (검색하면 나오는 정의는 빼기)
- "~란 무엇인가?" 식의 교과서 서두를 쓰지 않는다
- 불필요한 배경 지식 나열을 하지 않는다 (독자가 이미 해당 기술을 쓰고 있다고 가정)
- 참고자료 섹션에 링크를 5개 이상 나열하지 않는다
