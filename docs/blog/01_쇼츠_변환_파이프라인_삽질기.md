# "API 붙이면 끝"이 아니었다 — yt-dlp + Whisper + GPT 파이프라인 실전 구축기

## 들어가며

사이드 프로젝트로 요리 레시피 앱을 만들고 있다. 핵심 기능 중 하나가 **유튜브 쇼츠 URL을 붙여넣으면 구조화된 레시피로 변환해주는 것**이다.

설계는 단순해 보였다.

```
유튜브 URL → yt-dlp로 음성 추출 → Whisper로 STT → GPT로 구조화 → JSON 반환
```

"외부 API 3개 순서대로 호출하면 되겠지." 순진한 생각이었다. 커밋 로그를 보면 이 기능 하나에 6번의 버그 수정 커밋이 쌓여 있다. 각각이 "됐다고 생각했는데 프로덕션에서 터진" 케이스다.

## 라운드 1. ffmpeg를 못 찾는 yt-dlp

### 증상

로컬에서는 잘 돌아가는데, Docker 컨테이너에 올리니까 yt-dlp가 음성 추출을 못 한다.

```
ERROR: Postprocessing: ffprobe and ffmpeg not found.
```

### 원인

yt-dlp는 영상 다운로드 후 ffmpeg으로 오디오를 변환한다. 시스템 PATH에 ffmpeg이 있으면 자동으로 찾지만, Docker 이미지에서는 ffmpeg 바이너리 위치가 다를 수 있다. 그래서 `--ffmpeg-location` 옵션으로 명시적으로 알려줘야 한다.

### 함정

여기서 한 번 더 삽질했다. **`--ffmpeg-location`에는 바이너리 경로가 아니라 디렉토리 경로를 넘겨야 한다.**

```java
// 틀린 코드
"--ffmpeg-location", ffmpegPath  // /usr/bin/ffmpeg

// 맞는 코드
String ffmpegLocation = Path.of(ffmpegPath).getParent().toString();  // /usr/bin
"--ffmpeg-location", ffmpegLocation
```

yt-dlp 공식 문서에는 `--ffmpeg-location PATH`라고만 적혀 있어서, 바이너리 전체 경로를 넘기면 될 거라고 생각하기 쉽다. 실제로는 **ffmpeg과 ffprobe가 들어있는 디렉토리**를 가리켜야 한다.

### 또 다른 문제 — 대체 포맷

ffmpeg 경로를 잡아줬더니 이번엔 다른 문제가 생겼다. `.mp3`를 요청했는데 실제로는 `.webm` 파일이 생기는 경우가 있었다. yt-dlp는 원본 오디오 포맷에 따라 ffmpeg 변환을 건너뛰고 원본 그대로 저장하는 경우가 있다.

```java
// 요청한 .mp3가 없으면 대체 포맷 탐색
if (Files.exists(outputPath)) {
    return outputPath;
}
String baseName = outputPath.toString().replaceFirst("\\.[^.]+$", "");
for (String ext : new String[]{".webm", ".m4a", ".opus", ".ogg"}) {
    Path altPath = Path.of(baseName + ext);
    if (Files.exists(altPath)) {
        log.warn("yt-dlp produced {} instead of .mp3, renaming", ext);
        Files.move(altPath, outputPath);
        return outputPath;
    }
}
```

**교훈**: yt-dlp의 `--audio-format mp3` 옵션은 "가능하면 mp3로 변환해줘"이지, "반드시 mp3로 줘"가 아니다. 출력 파일의 존재 여부를 반드시 확인하고, 대체 포맷에 대한 폴백 로직이 필요하다.

## 라운드 2. GPT가 JSON을 마크다운으로 감싸서 보내줌

### 증상

GPT에게 "JSON으로만 응답하라"고 시스템 프롬프트에 명시했는데, 실제 응답이 이랬다.

```
```json
{
  "title": "김치볶음밥",
  "steps": [...]
}
```
```

당연히 Jackson `objectMapper.readValue()`에서 파싱 에러가 난다.

### 해결

GPT 모델(특히 Chat Completions API)은 시스템 프롬프트에서 "JSON만 반환하라"고 해도 마크다운 코드 블록으로 감싸는 경향이 있다. JSON mode를 강제하는 `response_format` 파라미터를 쓸 수도 있지만, 이미 배포된 프롬프트와의 호환성을 고려해서 후처리로 strip하는 방식을 택했다.

```java
String json = content.strip();
if (json.startsWith("```")) {
    json = json.replaceFirst("```(?:json)?\\s*", "");
    json = json.replaceFirst("\\s*```$", "");
}
return objectMapper.readValue(json, ShortsRecipeResult.class);
```

**교훈**: LLM의 출력은 프롬프트로 100% 통제되지 않는다. 특히 마크다운 코드블록 래핑은 매우 흔하므로, JSON 응답을 기대하는 곳에서는 반드시 strip 로직을 넣어두자.

## 라운드 3. yt-dlp 경고 메시지가 숫자 파싱을 깨뜨림

### 증상

영상 길이를 체크해서 3분 초과 영상을 거부하는 로직을 추가했다. 로컬에서 잘 되길래 배포했더니, 특정 영상에서 `NumberFormatException`이 발생했다.

### 원인

`yt-dlp --print duration` 명령의 출력이 항상 숫자 한 줄일 거라고 가정했는데, 실제로는 이런 식으로 나올 수 있다.

```
WARNING: [youtube] Unable to download API page: ...
WARNING: [youtube] Retrying (attempt 1 of 3)...
59.0
```

경고 메시지가 섞여서 `Double.parseDouble("WARNING: ...")`이 호출된 것이다.

### 해결

```java
// Before
double seconds = Double.parseDouble(output);

// After
ProcessBuilder pb = new ProcessBuilder(
    ytdlpPath, "--no-warnings", "--print", "duration", "--no-download", url
);
// ...
String lastLine = output.lines()
    .reduce((first, second) -> second)
    .orElse(output)
    .strip();
double seconds = Double.parseDouble(lastLine);
```

두 가지를 적용했다.
1. `--no-warnings` 플래그로 경고 출력 자체를 억제
2. 그래도 다른 출력이 섞일 수 있으니 **마지막 줄만 파싱**하는 방어 코드

**교훈**: CLI 프로세스의 stdout은 항상 예상대로 깨끗하지 않다. 경고, 진행률 표시, 디버그 정보가 섞일 수 있다. 파싱 전에 반드시 정제하고, 그래도 안 될 경우를 대비한 `NumberFormatException` 핸들링도 필요하다.

## 라운드 4. yt-dlp가 MCN 네트워크명을 채널명으로 반환

### 증상

쇼츠 변환 결과에 채널명을 표시하는데, 특정 채널의 영상에서 실제 채널명이 아니라 소속 네트워크 이름이 나왔다. 예를 들어 "백종원" 채널 영상인데 채널명이 다른 이름으로 표시되는 식이다.

### 원인

yt-dlp의 `--dump-json` 응답에는 `channel` 필드가 있는데, MCN(Multi-Channel Network) 소속 채널의 경우 이 필드에 네트워크명이 들어간다. 유튜브 내부적으로 channel과 uploader를 다르게 관리하기 때문이다.

### 해결

YouTube oEmbed API를 사용해서 정확한 채널 표시명을 가져오도록 변경했다.

```java
private String fetchChannelNameFromOEmbed(String videoUrl) {
    // shorts URL을 watch URL로 변환 (oEmbed는 watch URL만 지원)
    String watchUrl = videoUrl.replaceFirst("/shorts/", "/watch?v=");
    String encoded = URLEncoder.encode(watchUrl, StandardCharsets.UTF_8);
    String oembedUrl = "https://www.youtube.com/oembed?url=" + encoded + "&format=json";

    HttpResponse<String> response = httpClient.send(
        HttpRequest.newBuilder().uri(URI.create(oembedUrl)).GET().build(),
        HttpResponse.BodyHandlers.ofString()
    );

    if (response.statusCode() == 200) {
        JsonNode json = objectMapper.readTree(response.body());
        return getTextOrNull(json, "author_name");  // 실제 채널 표시명
    }
    return null;
}
```

oEmbed 실패 시에만 yt-dlp의 `channel` → `uploader` 순서로 fallback한다. 또 하나 주의점은 **oEmbed API는 `/shorts/` URL을 직접 지원하지 않아서** `/watch?v=` 형식으로 변환해야 한다는 것이다.

**교훈**: 크롤링/스크래핑 도구가 반환하는 메타데이터는 항상 의심해야 한다. 공식 API가 있다면 그쪽이 더 정확할 가능성이 높다. 단, 공식 API도 URL 형식 제약 같은 함정이 있으니 테스트를 충분히 해야 한다.

## 라운드 5. GPT가 필수 필드를 누락하고, 프롬프트 강화만으로는 부족

### 증상

코칭 모드에서 각 조리 단계에 `type`(active/wait)과 `durationSeconds`가 필요한데, GPT가 이 필드를 빼먹거나 `null`로 반환하는 경우가 있었다. 앱에서 타이머를 걸어야 하는데 시간 정보가 없으니 크래시가 났다.

### 해결 — 프롬프트 + 방어 코드 이중 방어

**1단계: 프롬프트 강화**
```
// Before (모호)
- durationSeconds는 대기 시간이 명확할 때만 설정, 없으면 null

// After (구체적)
- durationSeconds는 모든 단계에 반드시 정수값을 포함해야 한다 (null 불가)
- active 예시: "야채를 썬다" → 120, "양념장을 섞는다" → 60
- wait 예시: "10분간 끓인다" → 600, "30분간 재운다" → 1800
- 시간이 명시되지 않은 active 단계는 60~180 사이로 추정
```

**2단계: 기본값 방어 코드**
프롬프트를 아무리 강화해도 LLM은 지시를 100% 따르지 않는다. `withDefaults()` 메서드로 누락 시 기본값을 적용했다.

```java
public record ShortsRecipeStep(
    int stepNumber, String instruction, String type, Integer durationSeconds
) {
    public ShortsRecipeStep withDefaults() {
        String safeType = (type == null || type.isBlank()) ? "active" : type;
        Integer safeDuration = (durationSeconds == null) ? 60 : durationSeconds;
        return new ShortsRecipeStep(stepNumber, instruction, safeType, safeDuration);
    }
}
```

GPT 응답을 파싱한 직후에 모든 step에 일괄 적용한다.
```java
return objectMapper.readValue(json, ShortsRecipeResult.class).withStepDefaults();
```

**교훈**: LLM을 프로덕션에서 쓸 때는 **프롬프트 강화 + 코드 레벨 방어**를 반드시 이중으로 적용해야 한다. "프롬프트에 명시했으니까 괜찮겠지"는 위험한 가정이다.

## 라운드 6. 외부 API를 트랜잭션 안에서 호출하면 생기는 일

### 증상

쇼츠 변환이 몰리면 DB 커넥션 풀이 고갈되면서 다른 API(재료 목록, 레시피 상세 등)까지 느려지거나 타임아웃이 났다.

### 원인

초기 코드는 `@Transactional` 메서드 안에서 yt-dlp(최대 60초) → Whisper API(최대 60초) → GPT API(최대 30초)를 순차 호출하고 있었다. 즉 **한 번의 변환에 최대 2분 30초 동안 DB 커넥션을 물고 있는** 구조였다.

HikariCP 기본 풀 사이즈가 10개인데, 동시에 5명만 변환을 요청해도 절반이 장시간 점유된다.

### 해결

외부 API 호출은 트랜잭션 밖에서, DB 저장만 짧은 트랜잭션으로 분리했다.

```java
public ShortsConvertResponse convert(UUID userId, ShortsConvertRequest request) {
    // 캐시 조회 — 짧은 읽기 TX (자동)
    Optional<ShortsCache> cached = shortsCacheService.findByUrlHashAndModelVersion(...);
    if (cached.isPresent()) return ...; // 캐시 히트

    // 외부 API — TX 밖 (DB 커넥션 점유 없음)
    YtDlpResult ytResult = ytDlpService.fetchMetadataAndExtractAudio(url);     // ~60s
    String transcript = whisperService.transcribe(ytResult.audioPath());        // ~60s
    ShortsRecipeResult recipe = recipeStructurizer.structurize(transcript, ..); // ~30s

    // DB 저장만 — 짧은 TX (~수십ms)
    ShortsCache cache = transactionTemplate.execute(status -> {
        ShortsCache saved = shortsCacheService.save(new ShortsCache(...));
        historyRepository.save(new ShortsConversionHistory(userId, saved));
        conversionLogRepository.save(ShortsConversionLog.success(...));
        return saved;
    });

    return ShortsConvertResponse.of(cache, recipe, false);
}
```

추가로 **Rate Limiting**과 **동시성 제어**도 함께 적용했다.

```java
// 사용자별 한도
rateLimiter.checkUserLimit(userId);       // 5회/분, 50회/일

// 서버 전체 동시 처리 제한
rateLimiter.acquireConcurrentSlot();      // Semaphore(10)
try {
    // 외부 API 호출...
} finally {
    rateLimiter.releaseConcurrentSlot();  // 반드시 반환
}
```

**교훈**: 외부 API를 호출하는 로직에 `@Transactional`을 걸면, 응답 시간이 곧 DB 커넥션 점유 시간이 된다. 외부 호출이 느려지면 커넥션 풀이 고갈되어 전혀 관련 없는 API까지 영향받는다. **"트랜잭션은 DB 작업만 감싸야 한다"**는 원칙을 꼭 기억하자.

## 정리 — 외부 도구를 엮을 때 챙겨야 할 것들

| 구분 | 교훈 |
|------|------|
| CLI 프로세스 | stdout은 항상 더럽다. 경고·진행률이 섞인다. `--no-warnings` + 마지막 줄 파싱 |
| CLI 경로 | 옵션이 바이너리 경로인지 디렉토리 경로인지 반드시 확인 |
| CLI 출력 파일 | 요청한 확장자와 실제 확장자가 다를 수 있다. 존재 여부 확인 + 대체 포맷 폴백 |
| LLM 응답 | 마크다운 래핑 strip 필수. 필수 필드 누락 대비 기본값 방어 코드 필수 |
| LLM 프롬프트 | 모호한 지시("가능하면")보다 구체적 예시("야채를 썬다 → 120초")가 효과적 |
| 메타데이터 | 크롤링 도구의 데이터보다 공식 API가 정확하다 (단, URL 형식 제약 주의) |
| 트랜잭션 | 외부 API 호출을 트랜잭션 안에 넣지 말 것. DB 작업만 짧게 감쌀 것 |
| 동시성 | 비용이 큰 외부 호출에는 Rate Limit + Semaphore 동시 제한 |

6번의 삽질을 거치며 느낀 건, **"API 연동"은 호출 한 줄이 아니라 그 주변의 방어 코드가 본체**라는 것이다. Happy path는 1시간이면 만들지만, 실전에서 안정적으로 돌아가게 하려면 그 5배의 시간이 든다.
