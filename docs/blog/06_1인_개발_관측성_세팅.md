# 1인 개발 프로젝트에 관측성 세팅하기 — logback에서 Lighthouse 대시보드까지

## 개요

> 💡 Docker 컨테이너 재시작하면 로그가 전부 날아가고, 에러가 나도 원인 추적이 불가능했다. Spring Boot 모놀리스 하나에 logback 구조화 로깅 → MDC 기반 요청 추적 → AOP 성능 측정 → Micrometer 비즈니스 메트릭 → Vector → Lighthouse 대시보드까지, 처음부터 세팅한 과정을 정리했다.

## 출발점: 로그가 없는 운영

사이드 프로젝트를 EC2에 올리고 Docker로 운영 중이었다. 문제는 logback 설정을 아예 안 해뒀다는 것이다. `application.yml`에 `logging.level` 몇 줄 넣은 게 전부였고, 콘솔 출력만 하고 있었다.

Docker 컨테이너가 재시작되면 그동안의 로그가 전부 사라졌다. API 요청 로그도 없고, JWT 인증 실패 로그도 없고, 어떤 서비스 메서드가 느린지도 모른다. 쇼츠 변환이 500을 뱉어도 "뭔가 터졌다" 이상의 정보를 얻을 수 없었다.

결론부터 말하면, 아래 다섯 단계를 거쳐 해결했다.

| 단계 | 도구 | 역할 |
|------|------|------|
| 1 | logback-spring.xml | 환경별 로그 파일 분리 (4종) |
| 2 | RequestLoggingFilter + MDC | API 요청/응답 자동 기록 + 스레드 로컬 컨텍스트 |
| 3 | AOP PerformanceLoggingAspect | 서비스 메서드 실행 시간 자동 측정 |
| 4 | Micrometer @Timed + Counter | 비즈니스 메트릭 (추천 횟수, 쇼츠 변환 성공률 등) |
| 5 | Vector → Kafka → Lighthouse | 로그 수집 → 외부 모니터링 대시보드 |

## 1단계: logback 환경 분리 — 로그 파일 4개 구조

`logback-spring.xml`을 만들고, `springProfile`로 local/prod를 나눴다.

local은 단순하게 콘솔 DEBUG 출력만 한다. prod가 핵심인데, 로그 파일을 4개로 분리했다.

```xml
<!-- prod 프로필 로그 구조 -->
app.log    → INFO 이상, 일별 롤링, 30일 보관, 최대 1GB
error.log  → ERROR만, 일별 롤링, 90일 보관
sql.log    → Hibernate SQL 전용, 기본 OFF (디버깅 시만 활성화)
perf.log   → 서비스 메서드 성능 로그 전용
```

왜 4개로 나눴느냐. 하나에 다 몰아넣으면 에러 찾을 때 `grep`으로 뒤져야 한다. 파일을 분리해두면 `tail -f error.log`만으로 장애 상황 파악이 가능하다. sql.log는 평소에 꺼두고 "쿼리가 이상하다" 싶을 때만 켜는 용도다.

prod 콘솔은 WARN 이상만 출력한다. `docker logs`로 볼 때 INFO가 와르르 쏟아지면 정작 중요한 게 묻힌다.

```xml
<appender name="CONSOLE" class="ch.qos.logback.core.ConsoleAppender">
    <filter class="ch.qos.logback.classic.filter.ThresholdFilter">
        <level>WARN</level>
    </filter>
    ...
</appender>
```

롤링 정책은 `SizeAndTimeBasedRollingPolicy`를 썼다. 일별 롤링 + 파일 크기 50MB 제한 + 총 용량 캡. EC2 디스크가 무한하지 않으니 `totalSizeCap`을 반드시 걸어야 한다. 이걸 안 하면 로그가 디스크를 잡아먹어서 서버가 죽는 고전적인 사고가 난다.

## 2단계: RequestLoggingFilter — API 요청/응답 자동 기록

Spring의 `OncePerRequestFilter`를 상속해서 모든 API 요청을 자동으로 기록하게 만들었다.

```java
@Component
public class RequestLoggingFilter extends OncePerRequestFilter {

    @Override
    protected void doFilterInternal(...) {
        log.info("[REQUEST] {} {} - userId: {} ip: {}", method, uri, userId, clientIp);

        MDC.put("http_method", method);
        MDC.put("http_path", uri);
        MDC.put("client_ip", clientIp);

        long startTime = System.nanoTime();
        try {
            filterChain.doFilter(request, wrappedResponse);
        } finally {
            long elapsedMs = (System.nanoTime() - startTime) / 1_000_000;
            int status = wrappedResponse.getStatus();

            if (status >= 400) {
                log.warn("[ERROR] {} {} - {} - {}", method, uri, status, errorMessage);
            } else {
                log.info("[RESPONSE] {} {} - {} - {}ms", method, uri, status, elapsedMs);
            }
            MDC.clear(); // 정리
        }
    }
}
```

여기서 핵심은 **MDC(Mapped Diagnostic Context)**다. `MDC.put()`으로 넣은 값은 같은 스레드 내에서 실행되는 모든 로그에 자동으로 포함된다. 나중에 LogstashEncoder가 이 MDC 값을 JSON 필드로 뽑아주므로, Lighthouse에서 `http_method`, `http_path`, `response_time_ms` 같은 필드로 필터링/집계가 가능해진다.

`/actuator`, `/swagger-ui`, `/uploads` 같은 경로는 스킵한다. 이런 경로까지 로깅하면 노이즈만 늘어난다.

userId 추출은 SecurityContextHolder에서 꺼낸다. SecurityConfig에서 이 필터를 JwtAuthenticationFilter **뒤에** 등록해야 userId가 세팅된 상태에서 잡을 수 있다. 순서가 바뀌면 항상 "anonymous"로 찍힌다.

## 3단계: AOP 성능 측정

모든 서비스 메서드의 실행 시간을 자동으로 측정하는 Aspect를 만들었다.

```java
@Aspect
@Component
public class PerformanceLoggingAspect {

    private static final long SLOW_THRESHOLD_MS = 1000;

    @Around("execution(* com.picook.domain..service..*(..))")
    public Object logPerformance(ProceedingJoinPoint joinPoint) throws Throwable {
        long start = System.nanoTime();
        try {
            return joinPoint.proceed();
        } finally {
            long elapsedMs = (System.nanoTime() - start) / 1_000_000;
            String className = joinPoint.getTarget().getClass().getSimpleName();
            String methodName = joinPoint.getSignature().getName();

            if (elapsedMs >= SLOW_THRESHOLD_MS) {
                log.warn("[PERF] {}.{}() - {}ms [SLOW]", className, methodName, elapsedMs);
            } else {
                log.info("[PERF] {}.{}() - {}ms", className, methodName, elapsedMs);
            }
        }
    }
}
```

1초 이상 걸리면 WARN으로 찍는다. 이 로그는 `perf.log`에만 쌓인다(logback에서 `com.picook.global.aop` 로거를 `PERF_FILE` appender에만 연결). app.log에 성능 로그가 섞이지 않으니 각자 역할이 명확하다.

포인트컷이 `execution(* com.picook.domain..service..*(..))` — 모든 도메인 서비스 메서드를 잡는다. 새 서비스를 추가해도 자동으로 측정된다.

## 4단계: Micrometer 커스텀 메트릭

Actuator + Micrometer Prometheus 의존성을 추가하고, 비즈니스 로직에 메트릭을 심었다.

```java
// 추천 API — 실행 시간 자동 측정 + 호출 카운트 + 빈 결과 카운트
@Timed("picook.recommend.time")
public List<RecommendResponse> recommend(RecommendRequest request) {
    meterRegistry.counter("picook.recommend.requests").increment();
    ...
    if (responses.isEmpty()) {
        meterRegistry.counter("picook.recommend.empty_results").increment();
    }
    return responses;
}
```

```java
// 쇼츠 변환 — 성공/실패 카운트 분리
@Timed("picook.shorts.convert.time")
public ShortsConvertResponse convert(UUID userId, ShortsConvertRequest request) {
    ...
    meterRegistry.counter("picook.shorts.convert.success").increment();
    ...
} catch (BusinessException e) {
    meterRegistry.counter("picook.shorts.convert.failure").increment();
    ...
}
```

`@Timed`는 실행 시간 히스토그램을 자동으로 만들어준다. `MeterRegistry.counter()`는 단순 카운터. 이 두 가지만으로도 "추천 API가 몇 번 호출됐고 평균 몇 ms 걸리는지", "쇼츠 변환 성공률이 몇 %인지" 같은 걸 대시보드에서 바로 볼 수 있다.

Actuator 엔드포인트는 `/actuator/prometheus`로 노출했고, `MonitoringIpFilter`로 화이트리스트 IP만 접근 가능하게 막았다. 내부 모니터링 데이터가 외부에 노출되면 안 되니까.

## 삽질: Lighthouse에 에러 상세 정보가 안 올라간다

여기까지 세팅하고 Lighthouse 대시보드를 확인했는데, 에러가 발생한 요청의 `exception_class`와 `stack_trace`가 비어 있었다.

원인은 **MDC 저장 타이밍**이었다. RequestLoggingFilter의 `doFilter()`에서 예외가 컨트롤러까지 전파되지 않고, Spring의 `@ExceptionHandler`가 예외를 잡아서 정상 응답(400, 500)으로 바꿔 내려보낸다. 필터 입장에서는 "정상 응답"이 온 것이므로 catch 블록에 진입하지 않는다.

즉, 흐름이 이렇다:

```
RequestLoggingFilter → Controller → 예외 발생
                                    → @ExceptionHandler가 잡음
                                    → 400/500 JSON 응답 생성
                    ← 필터에는 "정상" 응답으로 돌아옴
                    → catch에 안 걸림 → MDC에 exception 정보 없음
```

**해결:** `GlobalExceptionHandler`에서 예외를 잡는 시점에 MDC에 직접 넣어주었다.

```java
@RestControllerAdvice
public class GlobalExceptionHandler {

    @ExceptionHandler(Exception.class)
    public ResponseEntity<ApiResponse<Void>> handleException(Exception e) {
        log.error("Unexpected error", e);
        putExceptionToMdc(e);  // ← 이 시점에 MDC 저장
        return ResponseEntity.status(HttpStatus.INTERNAL_SERVER_ERROR)
                .body(ApiResponse.error("INTERNAL_ERROR", "Internal server error"));
    }

    private void putExceptionToMdc(Exception e) {
        MDC.put("exception_class", e.getClass().getName());
        StringWriter sw = new StringWriter(1024);
        e.printStackTrace(new PrintWriter(sw));
        String stackTrace = sw.toString();
        if (stackTrace.length() > 4096) {
            stackTrace = stackTrace.substring(0, 4096) + "\n... truncated";
        }
        MDC.put("stack_trace", stackTrace);
    }
}
```

`GlobalExceptionHandler`가 MDC에 넣고, `RequestLoggingFilter`의 finally 블록에서 accessLog를 찍을 때 LogstashEncoder가 MDC 값을 JSON에 포함시킨다. MDC 정리도 필터의 finally에서 한다.

스택트레이스는 4096자로 잘랐다. Lighthouse 뒷단이 ClickHouse인데, 너무 긴 문자열을 넣으면 저장 효율이 떨어진다.

**결과: 성공.** Lighthouse에서 에러 발생 시 exception class와 스택트레이스를 바로 볼 수 있게 됐다.

## 5단계: Vector → Lighthouse

마지막 퍼즐은 로그를 서버 밖으로 보내는 것이다. logback이 JSON 파일을 만들고, Vector가 그 파일을 tail해서 Kafka로 보내고, Lighthouse가 수집한다.

```yaml
# vector.yaml
sources:
  picook_log:
    type: file
    include:
      - /var/log/picook/lighthouse.log

transforms:
  enrich:
    type: remap
    inputs: [picook_log]
    source: |-
      parsed = parse_json!(.message)
      .timestamp = parsed."@timestamp"
      .level = upcase(parsed.level)
      .http_method = upcase(parsed.http_method ?? "")
      .http_status = to_int(parsed.http_status) ?? 0
      .response_time_ms = to_int(parsed.response_time_ms) ?? 0
      .exception_class = parsed.exception_class ?? ""
      # ...

sinks:
  kafka_app:
    type: kafka
    inputs: [enrich]
    bootstrap_servers: ${LIGHTHOUSE_KAFKA_HOST}:9094
    topic: logs.app
    encoding:
      codec: json
```

logback 쪽에서는 `LogstashEncoder`를 사용해 lighthouse.log에 JSON 형태로 로그를 찍는다. MDC에 넣은 `http_method`, `http_status`, `response_time_ms` 같은 필드가 JSON 키로 바로 나온다.

Vector의 `remap` transform에서 타입 변환을 해준다. `http_status`가 문자열로 들어오니까 `to_int()`로 바꾸는 식이다. 이걸 안 하면 Lighthouse 쪽에서 숫자 필드를 문자열로 인식해서 집계가 안 된다.

Docker Compose에 Vector 컨테이너를 추가하고, picook 컨테이너의 로그 볼륨을 공유한다.

```yaml
vector:
  image: timberio/vector:0.45.0-alpine
  volumes:
    - ./vector/vector.yaml:/etc/vector/vector.yaml
    - picook_logs:/var/log/picook:ro
```

## 정리

| 구성요소 | 역할 | 핵심 포인트 |
|----------|------|------------|
| logback-spring.xml | 환경별 로그 파일 분리 | `totalSizeCap` 필수, prod 콘솔은 WARN만 |
| RequestLoggingFilter | API 요청/응답 자동 기록 | MDC로 스레드 컨텍스트 전파, Security 필터 순서 주의 |
| PerformanceLoggingAspect | 서비스 메서드 성능 측정 | 1초 이상이면 WARN, 전용 로그 파일 분리 |
| Micrometer | 비즈니스 메트릭 | `@Timed` + Counter로 핵심 지표 자동 수집 |
| LogstashEncoder + MDC | 구조화 JSON 로그 | MDC 값이 JSON 필드로 자동 포함 |
| Vector | 로그 수집 → Kafka 전송 | remap에서 타입 변환 필수 |
| GlobalExceptionHandler | MDC에 예외 정보 저장 | 필터가 아닌 핸들러에서 MDC에 넣어야 함 |

## 느낀 점

관측성을 "나중에"로 미루면 "나중"은 장애가 터졌을 때가 된다. 로그가 없으면 원인 추적이 불가능하고, 메트릭이 없으면 "느려진 것 같다"를 증명할 수 없다. 1인 개발이라 리소스가 부족하지만, 오히려 1인이니까 장애 대응을 자동화된 모니터링에 맡겨야 했다.

MDC 타이밍 문제는 의외로 문서에서 잘 안 다루는 부분이었다. Spring의 `@ExceptionHandler`가 예외를 삼켜버리기 때문에 필터 레벨에서는 "에러가 없는 것"처럼 보인다는 걸, 직접 로그를 보고 나서야 이해했다.
